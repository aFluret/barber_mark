Цель: обеспечить безопасную миграцию и снизить риск утечек секретов.

### 6.1. Секреты и доступы
- не коммитить `.env`, приватные ключи, токены, конфиги секретов в git;
- использовать GitHub Secrets/Variables для CI;
- принцип наименьших привилегий: отдельные токены/ключи на операции (read/write по необходимости).

### 6.2. Перенос CI/CD
- проверить, что workflow-файлы используют переменные окружения, а не захардкоженные значения;
- после миграции убедиться, что старые секреты не "текут" в новый контур.

### 6.3. Проверка на утечки
- перед пушем в новый репозиторий: grep/rg (или специализированные инструменты) на наличие токенов/ключей в истории и конфигурациях;
- использовать GitHub secret scanning (если включено) и локальные проверки.

# Project: Telegram-бот для записи в барбершоп

## 1. Цели проекта

### Бизнес-цели
- Сократить время записи клиента до 10 секунд.
- Снизить ручную нагрузку на барбера по обработке записей.
- Обеспечить прозрачный и актуальный список будущих записей.

### Продуктовые цели MVP
- Простой сценарий self-service записи в Telegram.
- Минимальный набор действий без сложной навигации.
- Надежные напоминания за 24 часа и за 2 часа.

### Нефункциональные цели
- Предсказуемое поведение и стабильность.
- Безопасное хранение персональных данных.
- Поддерживаемая архитектура с ясным разделением ответственности.

---

## 2. Область MVP

### Включено в MVP
- Онбординг клиента: `/start`, согласие на передачу контакта, ввод имени.
- Запись на услугу: выбор даты, выбор слота, подтверждение.
- Просмотр и отмена текущей записи клиентом.
- Команды администратора: `/today`, `/tomorrow`, `/all`.
- Настройка рабочего графика и автоматическая генерация слотов.
- Напоминания клиентам (24ч и 2ч до визита).

### Исключено из MVP
- Сложные вложенные меню.
- Ручной ввод времени.
- Дополнительные поля (комментарии, услуги).
- Маркетинговая реактивация и расширенная аналитика.

---

## 3. Архитектура (логическая)

### 3.1 Компоненты
- **Telegram Adapter Layer**: обработка входящих апдейтов и команд.
- **Application Layer (Use Cases)**:
  - Регистрация пользователя.
  - Создание записи.
  - Отмена записи.
  - Получение списков записей для администратора.
  - Планирование и отправка напоминаний.
- **Domain Layer**:
  - Сущности: User, Appointment, WorkSchedule, TimeSlot.
  - Правила: недоступность занятых слотов, мгновенное освобождение при отмене.
- **Infrastructure Layer**:
  - Репозитории БД.
  - Планировщик задач (APScheduler).
  - Конфигурация, логирование, миграции.

### 3.2 Потоки данных
1. Пользователь выбирает слот через бота.
2. Сервис получает список доступных слотов (с учетом графика и занятости).
3. При подтверждении создается запись со статусом `confirmed`.
4. Создаются задачи напоминаний (24ч/2ч).
5. При отмене запись переводится в `cancelled`, слот сразу становится доступным.

### 3.3 Принципы проектирования
- **SOLID**: изоляция use case-логики от Telegram API и БД.
- **KISS**: только необходимые сценарии MVP.
- **DRY**: единый сервис генерации слотов и валидации времени.
- **Fail-safe defaults**: безопасные значения по умолчанию в конфигурации.

## 4. Технологический стек

### Рекомендованный стек
- Язык: **Python 3.12+**
- Telegram framework: **aiogram 3.x**
- База данных MVP: **Supabase (PostgreSQL)**
- Планировщик: **APScheduler**
- Миграции: SQL-миграции в PostgreSQL/Supabase
- Конфиг: **Pydantic Settings + .env**
- Тестирование: **pytest**

### Причины выбора
- Быстрый старт для MVP.
- Низкий порог сопровождения.
- Четкая дорожка масштабирования до PostgreSQL.

## 6. Модель данных

### Таблица `users`
- `user_id` (PK, bigint): Telegram ID
- `phone` (text, not null)
- `name` (text, not null)
- `created_at` (timestamp, not null)

### Таблица `appointments`
- `id` (PK, integer/uuid)
- `user_id` (FK -> users.user_id, not null)
- `date` (date, not null)
- `time_slot` (time, not null)
- `status` (text/check: confirmed|cancelled|completed)
- `created_at` (timestamp, not null)

### Таблица `work_schedule`
- `id` (PK)
- `weekdays` (int[]): python weekday для БД/бота (Пн=0 ... Вс=6) — формируется из `/set_schedule`
- `start_time` (time, not null)
- `end_time` (time, not null)
- `created_at` (timestamp, not null)

### Рекомендуемые ограничения
- `unique(date, time_slot)` для записей со статусом `confirmed` (через частичный индекс/проверку в сервисе).
- Индекс для частых запросов: `(date, status)`.
- Валидация времени кратно шагу слота.

## 7. Безопасность и приватность

### Требования
- Телефон сохраняется только после явного действия пользователя (кнопка контакта).
- Данные не передаются третьим сторонам.

### Практические меры
- Проверка, что переданный контакт принадлежит инициатору (`contact.user_id == from_user.id`).
- Хранение токена бота только в переменных окружения.
- Маскирование телефона в логах.
- Минимизация логирования персональных данных.
- Ротация токена при компрометации.

## 8. Производительность и надежность

### Целевые показатели MVP
- Время ответа бота на действие пользователя: < 1.5 сек.
- Формирование доступных слотов: < 300 мс при дневном диапазоне.
- Стабильная отправка напоминаний с допустимой задержкой до 1 минуты.

### Технические меры
- Кэширование рабочей сетки слотов на день.
- Транзакционное создание записи для исключения гонок.
- Идемпотентность отправки напоминаний (через маркер отправки).
- Retry при временных ошибках Telegram API.
- Расчет времени напоминаний в локальном timezone барбера (по умолчанию `Europe/Minsk`, UTC+3).

## 12. Риски и план снижения

- **Риск:** коллизии записи в один слот при конкурентных запросах.  
  **Снижение:** транзакции + ограничения уникальности.
- **Риск:** пропуск уведомлений из-за перезапуска сервиса.  
  **Снижение:** восстановление задач из БД при старте.
- **Риск:** утечка персональных данных в логах.  
  **Снижение:** маскирование и аудит логирования.
- **Риск:** рост нагрузки после MVP.  
  **Снижение:** Supabase/PostgreSQL готов к масштабированию без смены бизнес-логики.
