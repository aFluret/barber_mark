# Техническое задание (расширенное, все ответы из `qa.md` — положительные)

## 1. Основа
Данный документ расширяет `TZ_MVP.md` и фиксирует, что на каждый вопрос из `docs/qa.md` получен положительный ответ. Все дополнительные возможности включаются в объем работ.

## 2. Подтвержденные решения по `qa.md`

### 2.1 Бизнес-правила записи
- Разрешена более чем одна активная запись у одного клиента.
- Разрешена запись "день в день" (минимальный горизонт: не позже чем за 1 час до слота).
- Период отображения доступных дат: 30 дней вперед.

### 2.2 Рабочий график барбера
- Поддерживаются исключения в графике (праздники, отпуск, разовые выходные).
- Поддерживаются разные интервалы по дням недели.
- Шаг слота редактируется через админ-интерфейс.

### 2.3 Роли и доступ
- Поддерживается список администраторов (`admin_user_id[]`).
- Добавляется команда управления администраторами без деплоя.

### 2.4 Уведомления
- Напоминания считаются в локальном часовом поясе барбера.
- Реализуются повторные попытки отправки при ошибках Telegram.
- Добавляется дополнительное напоминание за 30 минут (post-MVP становится частью расширенного объема).

### 2.5 Данные и безопасность
- Вводится срок хранения персональных данных и удаление по запросу клиента.
- Фиксируется юридическое согласие на обработку данных (текст политики/оферты).
- В продакшене применяется безопасный уровень логирования с маскированием персональных данных.

### 2.6 Эксплуатация и релиз
- MVP запускается в Docker на VPS.
- Используется CI/CD pipeline.
- Настраивается резервное копирование БД по расписанию.

## 3. Дополнительные функциональные требования

### 3.1 Расписание и календарная логика
- Поддержка базового недельного расписания + исключений по датам.
- Поддержка изменения шага слота администратором.
- Автоматический пересчет доступных слотов при обновлении графика.

### 3.2 Управление администраторами
- Команды:
  - `/admins` — список администраторов.
  - `/admin_add <telegram_id>` — добавить администратора.
  - `/admin_remove <telegram_id>` — удалить администратора.
- Все команды доступны только текущим администраторам.

### 3.3 Уведомления и надежность отправки
- Типы напоминаний: 24ч, 2ч, 30м.
- Retry-политика: до 3 попыток при временных ошибках.
- Идемпотентность задач напоминания для исключения дублей.

### 3.4 Политика данных
- Настраиваемый срок хранения персональных данных.
- Команда удаления данных пользователя по запросу.
- Хранение факта согласия и версии текста политики.

### 3.5 Эксплуатация
- Docker-образ приложения.
- CI/CD с проверкой линтеров и тестов.
- План резервного копирования с проверкой восстановления.

## 4. Нефункциональные требования (расширенные)
- Сохранить целевой UX записи за 10 секунд для типового сценария.
- Обеспечить отказоустойчивость отправки напоминаний.
- Обеспечить аудит действий администраторов.
- Гарантировать масштабируемость на переход к PostgreSQL без изменения бизнес-логики.

## 5. Изменения в модели данных

### 5.1 Новые/расширенные сущности
- `admin_users` — список администраторов.
- `work_schedule` — базовый график (дни, интервалы, шаг).
- `schedule_exceptions` — исключения по датам.
- `reminder_jobs` — запланированные и отправленные напоминания.
- `consents` — фиксация согласий пользователя.
- `deletion_requests` — запросы на удаление данных.

### 5.2 Изменения в `appointments`
- Поддержка нескольких активных записей у одного пользователя.
- Индексы для ускорения выборок по дате, статусу и пользователю.

## 6. Архитектурные требования
- Сервис расписаний должен поддерживать регулярные правила и исключения.
- Сервис уведомлений должен быть идемпотентным и устойчивым к перезапускам.
- Репозитории и use case-слой должны быть независимы от конкретной БД.
- Логи должны быть структурированными и безопасными.

## 7. План реализации
1. Реализовать MVP-контур из `TZ_MVP.md`.
2. Добавить многопользовательский admin-контур.
3. Добавить расширенную календарную модель и исключения.
4. Расширить уведомления и retry-логику.
5. Добавить политику хранения/удаления персональных данных.
6. Подготовить Docker, CI/CD и backup-процессы.

## 8. Критерии приемки (расширенные)
- Все критерии из MVP выполняются.
- Работают исключения графика и гибкий шаг слота.
- Работает управление администраторами через команды.
- Напоминания приходят 24ч/2ч/30м с retry при ошибках.
- Есть подтверждаемое резервное копирование и восстановление БД.
