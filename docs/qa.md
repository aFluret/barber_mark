# qa.md

## 1. Вопросы по миграции GitHub (обязательные для старта)
## 1.1 Новый репозиторий и доступ
1. Какой точный URL нового GitHub-репозитория (`NEW_REPO_URL`) и чьей организации он принадлежит?
2. Как должен осуществляться доступ при работе с репозиторием:
   - `SSH (deploy key)`
   - `HTTPS + PAT`
   - `GitHub App`
3. Должны ли быть разные разрешения для CI и для разработчиков (минимальные привилегии)?
4. Будет ли новый репозиторий приватным/публичным, и есть ли ограничения по окружениям (`staging/prod` защищённые правила)?

## 1.2 CI/CD и workflow
1. Используются ли GitHub Actions сейчас? Если да — в каком наборе workflow?
2. Есть ли deploy-стратегия (например, статическая раздача, Docker registry, сервер через SSH и т.п.)?
3. Где хранятся секреты:
   - `Settings -> Secrets and variables` (GitHub)
   - или внешняя система (например, Cloud provider secrets)?
4. Нужно ли переносить `environments`, `required reviewers`, `branch protection rules`?

## 1.3 Репозиторий-источник
1. Нужно ли архивировать старый репозиторий, удалять или только отвязать `remote`?
2. Есть ли открытые `issues/PRs/milestones` в старом репозитории: их нужно мигрировать вручную или оставить?
3. Есть ли `submodules`/`git-lfs`/кастомные hooks, которые нельзя терять?

## 2. Вопросы по рефакторингу (после заполнения docs/TZ_MARK)
1. Какой основной результат рефакторинга ожидается: скорость разработки, надежность, безопасность, снижение техдолга, изменение функциональности?
2. Какой целевой стек/архитектура предполагаются (язык, фреймворк, шаблоны, подходы к слоям)?
3. Есть ли ограничения по времени/рискам: можно ли "big-bang", или нужен поэтапный rollout?
4. Какие нефункциональные требования важны:
   - время ответа
   - пропускная способность
   - требования к логированию/наблюдаемости
5. Какие тесты должны быть обязательными:
   - unit/integration/e2e
   - какие критерии покрытия/качества
6. Есть ли требования по безопасности на уровне кода:
   - SAST/Dependency scanning
   - threat modeling для ключевых потоков

## 3. Обязательные уточнения по TZ_MARK
1. Правила overlap/границ: если новая запись начинается ровно в момент `end_time` существующей записи (интервалы касаются границей), считаем это допустимым или запрещаем?
2. Правила overlap/границ: если новая запись заканчивается ровно в `start_time` существующей записи — допустимо?
3. Обед `14:00–15:00`: запрещаем любой интервал, который пересекается с обедом хотя бы на одну минуту (включая случаи “начинается до обеда и заканчивается внутри/после” и “начинается внутри и заканчивается после”) — или действует более узкое правило “нельзя начинать внутри обеда”?
4. Рабочий день `10:00–20:00`: можно ли назначать запись так, чтобы `end_time == 20:00`?
5. Шаг слота 30 минут: старт слота должен быть кратен 30 минутам относительно `00` (то есть 10:00, 10:30, ...) или относительно `start_work_time` (например, если start_work_time изменится в админке)?
6. Услуга “Отец + Сын”: длительность строго `120 минут` — подтверждаем, или нужно отдельное уточнение/другая длительность?
7. Админка “Записи на другой день”: сколько ближайших рабочих дней показывать в списке (например, 3, 5 или все на некотором горизонте)? И начинать показ с “завтра” или с “следующего рабочего дня” относительно выбранного?
8. Админка “другой день”: показываем ближайшие дни без воскресенья — подтверждаем?
9. Админка: после выбора дня выводим только `confirmed` записи или учитывать также отмененные/другие статусы?
10. Один клиент: разрешаем ли несколько активных записей у одного пользователя одновременно (в текущем коде запрещено), или это правило нужно изменить под TZ_MARK?

# Вопросы по архитектуре и требованиям

## 1) Бизнес-правила записи
- Нужна ли возможность иметь более одной активной записи у одного клиента?
- Разрешаем ли запись "день в день" и до какого минимального горизонта (например, не позже чем за 1 час)?
- Какой допустимый период для отображения доступных дат (7, 14, 30 дней)?

## 2) Рабочий график барбера
- График один общий или требуется поддержка исключений (праздники, выходные, отпуск)?
- Нужны ли отдельные правила для конкретных дней недели (разные интервалы по будням/выходным)?
- Шаг слота (30 минут) всегда фиксированный или должен редактироваться в админ-интерфейсе?

## 3) Роли и доступ
- Сколько администраторов поддерживается: один барбер или список `admin_user_id`?
- Нужна ли команда для управления списком админов без деплоя?

## 4) Уведомления
- В каком часовом поясе должны считаться напоминания (локальное время барбера или UTC с конвертацией)?
- Требуется ли подтверждение доставки/повторная попытка при ошибке отправки Telegram?
- Нужны ли дополнительные напоминания (например, за 30 минут) в post-MVP?

## 5) Данные и безопасность
- Нужен ли срок хранения персональных данных и политика удаления по запросу клиента?
- Нужна ли фиксация юридического согласия на обработку контакта (текст оферты/политики)?
- Какой уровень детализации логов допустим в продакшене?

## 6) Эксплуатация и релиз
- Где планируется запуск MVP: VPS, Docker, serverless?
- Нужен ли CI/CD на этапе MVP или достаточно ручного деплоя?
- Требуется ли резервное копирование БД и с какой периодичностью?
