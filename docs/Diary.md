# Diary

## [2026-03-23] Инициализация архитектурного контура

### Наблюдения
- Проект ориентирован на быстрый MVP, где главный KPI — запись за 10 секунд.
- Критичны простота UI, отсутствие когнитивной нагрузки и минимальное число шагов.
- Основной риск — коллизии слотов и неконсистентность при отменах/напоминаниях.

### Решения
- Зафиксирована слоистая архитектура (adapter -> application -> domain -> infrastructure).
- Для MVP выбран Supabase (PostgreSQL) как целевая БД.
- Отложенные уведомления решено реализовать через APScheduler с восстановлением задач при старте.
- В документации закреплены принципы SOLID/KISS/DRY и критерии готовности.

### Проблемы
- Не определен окончательный формат хранения рабочего расписания (в БД или в конфиге).
- Не уточнены требования к часовому поясу и режиму летнего времени.
- Не зафиксирован окончательный перечень администраторов (один или несколько user_id).

## [2026-03-23] Подготовка двух технических заданий

### Наблюдения
- Появилась необходимость формально разделить MVP-объем и расширенный объем.
- Для расширенного ТЗ принята гипотеза: по всем пунктам `qa.md` ответы положительные.

### Решения
- Создано отдельное ТЗ для MVP: `docs/TZ_MVP.md`.
- Создано отдельное расширенное ТЗ: `docs/TZ_EXTENDED_ALL_POSITIVE.md`.
- В расширенном варианте зафиксированы дополнительные требования по админам, расписанию, уведомлениям, политике данных и эксплуатации.

### Проблемы
- Требуются окончательные подтверждения параметров (например, горизонт записи, среда развертывания), чтобы перейти от гипотез к утвержденным требованиям.

## [2026-03-23] Старт реализации MVP-каркаса

### Наблюдения
- Пользователь подтвердил переход к реализации согласно `docs/TZ_MVP.md`.
- На текущем шаге приоритет — создать фактическую структуру кода и базовые модули без перегрузки логикой.

### Решения
- Сначала актуализируется документация, затем создается каркас `src/` и `tests/`.
- Базовая реализация выполняется на Python + aiogram 3.x с модульной структурой слоев.

### Проблемы
- Полная бизнес-логика записи и интеграция БД будут реализованы следующими шагами; на этом этапе формируется foundation.

## [2026-03-23] Реализация MVP foundation: структура и базовые модули

### Наблюдения
- Создана фактическая структура `src/` и `tests/` согласно `TZ_MVP.md`.
- Добавлены базовые хендлеры, сервисы, конфиг, модели и стартовые тесты.

### Решения
- Зафиксирован единый вход приложения в `src/main.py`.
- Базовая архитектура отражает разделение на bot/app/domain/infra.
- Добавлены `requirements.txt` и `.env.example` для ускорения последующей сборки.

### Проблемы
- Временная демо-логика в хендлерах и сервисах требует замены на реальную БД- и FSM-логику.
- Не настроены линтеры и pre-commit hooks (задача остается открытой).

## [2026-03-23] Реализация реальной бизнес-логики записи (Supabase + FSM)

### Наблюдения
- Для MVP выбран Supabase (PostgreSQL), а не SQLite.
- Сценарий записи удобно реализовывать через FSM: регистрация (контакт+имя) и затем booking (дата/время/подтверждение).

### Решения
- Добавлены FSM states `RegistrationStates` и `BookingStates`.
- `/start` реализован как регистрационный сценарий с явным согласием на контакт.
- Сценарий `📅 Записаться` переведен на inline-клавиатуры с выбором даты и слота.
- Добавлена бизнес-логика подтверждения и отмены через `BookingService` и Supabase-репозитории.
- `/today`, `/tomorrow`, `/all` реализованы через чтение из `appointments` и подтягивание `name/phone` из `users`.

### Проблемы
- Возможны гонки при одновременной записи в один слот: для полной защиты нужен уникальный constraint в БД (например, на (date,time_slot) для `status='confirmed'`) или RPC-операция.
