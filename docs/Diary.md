# Diary

## [2026-03-23] Инициализация архитектурного контура

### Наблюдения
- Проект ориентирован на быстрый MVP, где главный KPI — запись за 10 секунд.
- Критичны простота UI, отсутствие когнитивной нагрузки и минимальное число шагов.
- Основной риск — коллизии слотов и неконсистентность при отменах/напоминаниях.

### Решения
- Зафиксирована слоистая архитектура (adapter -> application -> domain -> infrastructure).
- Для MVP выбран Supabase (PostgreSQL) как целевая БД.
- Отложенные уведомления решено реализовать через APScheduler с восстановлением задач при старте.
- В документации закреплены принципы SOLID/KISS/DRY и критерии готовности.

### Проблемы
- Не определен окончательный формат хранения рабочего расписания (в БД или в конфиге).
- Не уточнены требования к часовому поясу и режиму летнего времени.
- Не зафиксирован окончательный перечень администраторов (один или несколько user_id).

## [2026-03-23] Реализация реальной бизнес-логики записи (Supabase + FSM)

### Наблюдения
- Для MVP выбран Supabase (PostgreSQL), а не SQLite.
- Сценарий записи удобно реализовывать через FSM: регистрация (контакт+имя) и затем booking (дата/время/подтверждение).

### Решения
- Добавлены FSM states `RegistrationStates` и `BookingStates`.
- `/start` реализован как регистрационный сценарий с явным согласием на контакт.
- Сценарий `📅 Записаться` переведен на inline-клавиатуры с выбором даты и слота.
- Добавлена бизнес-логика подтверждения и отмены через `BookingService` и Supabase-репозитории.
- `/today`, `/tomorrow`, `/all` реализованы через чтение из `appointments` и подтягивание `name/phone` из `users`.

### Проблемы
- Возможны гонки при одновременной записи в один слот: для полной защиты нужен уникальный constraint в БД (например, на (date,time_slot) для `status='confirmed'`) или RPC-операция.

## [2026-03-23] Обработка unique_violation при конкурентной записи

### Наблюдения
- После создания уникального индекса `appointments_confirmed_slot_unique` при конкурентной записи возможна ошибка вставки вместо “предварительной проверки”.

### Решения
- В `src/infra/db/repositories/appointments_repository.py` добавлен перехват ошибок вставки.
- Ошибка уникальности (unique_violation/23505/duplicate key/имя индекса) преобразуется в `SlotUnavailableError`, который в UI соответствует состоянию “слот уже занят”.

### Проблемы
- Для полной уверенности нужны интеграционные тесты/ручной гоняющий сценарий.

## [2026-03-23] Исправление insert().select в supabase-py

### Наблюдения
- При нажатии кнопки “✅ Подтвердить” бот падал с `AttributeError: 'SyncQueryRequestBuilder' object has no attribute 'select'`.
- Причина: supabase-py 2.x не поддерживает цепочку `.insert(...).select(...)`.

### Решения
- В `src/infra/db/repositories/appointments_repository.py` удалена `.select()` после `insert`.
- После вставки запись того же слота выбирается отдельным запросом `select(...)`.
- Добавлена “страховка” в UI-обработчик `confirm_or_back`, чтобы не было тишины при внутренних ошибках.

### Проблемы
- Нужна финальная проверка end-to-end сценария в Telegram после перезапуска polling.
